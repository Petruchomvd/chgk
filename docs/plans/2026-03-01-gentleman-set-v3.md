# Джентльменский набор v3: план повышения точности

**Дата:** 2026-03-01  
**Статус:** План (к реализации после обсуждения)

## 1. Цель

Сделать «Джентльменский набор» полезным для подготовки к ЧГК:

- показывать частые **осмысленные маркеры** (а не случайные слова);
- стабильно раскладывать ответы по 6 категориям:
  - `Люди`
  - `Места`
  - `Произведения`
  - `Наука`
  - `Выражения`
  - `Числа`
- уменьшить число ложных попаданий и спорных кейсов.

## 2. Текущее состояние и ключевые проблемы

### 2.1 Что не устраивает в результате

- есть очевидные ошибки типа:
  - `палач`, `нимб` в `Люди`;
  - `фейерверк` в `Места`;
  - спорные `Произведения`;
- по ряду категорий низкое покрытие (особенно `Выражения`, `Наука`);
- в NER-вкладке «Организации» много шумных сущностей;
- пользователь выбирает `top-N`, но фактически видит меньше элементов (из-за ограничений данных/фильтров).

### 2.2 Корневые причины

1. Категоризация выполняется по одному `answer` без контекста вопроса.
2. 6-категорийная схема не покрывает весь спектр ответов (много «объектов/символов»).
3. Валидация категорий опирается на хрупкие правила/словари и даёт баланс «слишком мягко/слишком строго».
4. Нет формальной метрики качества (`precision@K`, error buckets) для управляемой итерации.
5. NER на коротких ответах (особенно ORG) шумный по природе.

## 3. Целевые метрики (Definition of Success)

После v3 считаем результат годным, если:

1. `precision@20` по каждой из 6 категорий >= 0.85 на ручной валидации.
2. В `Выражениях` нет мусора типа `да/нет/не/ничего` в top-20.
3. В `Науке` top-20 содержит преимущественно термины/законы/принципы.
4. В `Произведениях` есть репрезентативный набор разных типов (книги/фильмы/картины/игры/музыка).
5. UI честно показывает, когда фактический объём категории меньше выбранного `top-N`.

## 4. Основная стратегия v3 (рекомендуемая)

Использовать **гибридный pipeline**:

1. Детеминированные правила для очевидных кейсов (числа, явные паттерны).
2. Словари/курируемые списки для канонических ЧГК-маркеров.
3. LLM только для сложного остатка.
4. Абстейн (uncategorized), если уверенности нет.
5. Ручная доразметка наиболее частых ошибок (active learning).

## 5. План реализации (2 спринта)

## Спринт A — стабилизация качества

### A1. Ввести baseline-оценку

- Создать `eval`-выборку 800-1500 самых частых ответов.
- Разметить вручную в 6 классов + `вне таксономии`.
- Добавить отчёт по метрикам:
  - `precision@20` по каждой категории;
  - `coverage`;
  - топ-ошибки по типам.

### A2. Улучшить нормализацию и кандидаты

- Расширить нормализацию `answer`:
  - варианты пунктуации/кавычек/дефисов;
  - `ё/е`, пробелы, регистр;
  - базовые алиасы.
- Добавить `pack_diversity` в ранжирование:
  - использовать не только `count`, но и `unique_packs`.

### A3. Укрепить категориальные валидаторы

- Разделить валидаторы на 3 типа:
  - `hard allow` (явно разрешённые);
  - `hard deny` (явно запрещённые);
  - `soft rules` (маркеры).
- Пересмотреть спорные правила:
  - для `Люди` убрать слабые эвристики;
  - для `Места` и `Произведения` расширить/уточнить признаки;
  - для `Выражения` добавить curated-фразы.

### A4. ORG в NER перевести в «вспомогательный» статус

- Дополнительно фильтровать ORG (пересечения с PER/LOC, стоп-список шумных шаблонов).
- В UI пометить, что NER/ORG — аналитический вспомогательный слой, не источник истины для Gentleman Set.

## Спринт B — повышение recall и устойчивости

### B1. Контекстная LLM-категоризация

- Для неоднозначных ответов передавать в LLM:
  - `answer`;
  - 1-3 примера вопросов, где этот ответ встречается.
- Ожидаемый эффект:
  - меньше ложных переносов в `Люди/Места`;
  - лучшее качество `Выражения` и `Наука`.

### B2. Активное обучение (итеративно)

- На каждом цикле выбирать:
  - top-frequency `uncategorized`;
  - top-frequency ошибки из ручной проверки.
- После ручной валидации обновлять:
  - curated lists;
  - deny/allow-правила;
  - few-shot блоки для LLM.

### B3. Перегенерация полного набора

- Запускать категоризацию не на 500, а на 1000+ кандидатов.
- Публиковать в JSON прозрачные мета-поля:
  - `source_candidates`;
  - `categorized`;
  - `uncategorized`;
  - `min/max frequency`;
  - `pack_diversity`.

## 6. Альтернативные архитектуры

## Вариант 1. KB-first + LLM fallback

Суть:

- сначала проверка по словарям/базам сущностей (локальным/внешним),
- потом LLM на остаток.

Плюсы:

- высокая стабильность для `Люди/Места/Произведения`.

Минусы:

- нужна поддержка словарей и синхронизация.

## Вариант 2. Weak supervision (labeling functions)

Суть:

- набор функций-разметчиков (regex/NER/словарь/LLM-сигнал),
- объединение в итоговую метку.

Плюсы:

- прозрачность источников решения.

Минусы:

- выше инженерная сложность на старте.

## Вариант 3. Fine-tune small classifier

Суть:

- обучить отдельный классификатор на размеченной выборке `(question + answer)`.

Плюсы:

- дешевле и стабильнее в массовом прогоне.

Минусы:

- нужен качественный датасет и цикл дообучения.

## 7. Приоритетный roadmap

1. `A1` baseline/eval (обязательно).
2. `A3` правила и валидаторы.
3. `A2` pack diversity + нормализация.
4. `B1` контекстная LLM для сложных кейсов.
5. `B2` active learning итерациями.

## 8. Что менять в коде

- `scripts/analyze_answers.py`
  - нормализация ответов;
  - pack diversity;
  - улучшение ORG-фильтра.
- `scripts/categorize_gentleman.py`
  - строгие валидаторы категорий;
  - контекстный режим категоризации;
  - логика абстейна и метрики.
- `dashboard/app.py`
  - честные подписи `top-N`;
  - UI-индикаторы coverage/uncategorized;
  - маркировка NER как вспомогательного режима.

## 9. Риски

- Слишком жёсткие правила => рост `uncategorized`.
- Слишком мягкие правила => шум в категориях.
- Без eval-набора любые изменения будут «на глаз».

## 10. Критерии приёмки v3

1. Есть reproducible eval-скрипт и baseline-отчёт.
2. `precision@20` по категориям задокументирован и вырос относительно baseline.
3. Визуальные ошибки первого уровня (из пользовательского фидбека) устранены.
4. JSON и UI прозрачно показывают качество/покрытие, а не только «красивый топ».
