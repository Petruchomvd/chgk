# Улучшение проекта CHGK: классификация, скорость, аналитика

## Контекст

Проект классифицирует ~37,500 вопросов ЧГК по 14 категориям/52 подкатегориям через LLM.
Цель — понять, какие тематики встречаются чаще всего, чтобы эффективнее готовиться к турнирам.

**Текущее состояние:**
- Скрапинг завершён: 37,486 вопросов, 806 пакетов
- Классифицировано: **50 из 37,486 (0.13%)** — только бенчмарк-набор
- Лучшая модель: Qwen 14B двухэтапный — **88% accuracy**, 17с/вопрос
- Оценочное время полной классификации: **~7.4 дней** непрерывно

**Три оси улучшений:**
1. Скорость — закончить классификацию быстрее
2. Качество — повысить точность и надёжность
3. Аналитика — получить actionable инсайты из результатов

## 1. СКОРОСТЬ: ускорить классификацию в 2-3 раза

### 1.1 Параллельные запросы к Ollama (ThreadPoolExecutor)

**Проблема:** Сейчас код в `classifier/runner.py` (строка 182) обрабатывает вопросы строго последовательно. Каждый вопрос — 2-3 синхронных вызова `ollama.chat()`. GPU RTX 3050 простаивает между отправкой ответа и получением нового запроса.

**Решение:** Ollama поддерживает параллельные запросы (параметр `OLLAMA_NUM_PARALLEL`). Добавить `concurrent.futures.ThreadPoolExecutor` с 2 воркерами:

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def run_classification_parallel(questions, *, workers=2, ...):
    with ThreadPoolExecutor(max_workers=workers) as pool:
        futures = {pool.submit(_classify_one, q): q for q in questions}
        for future in as_completed(futures):
            q = futures[future]
            topics = future.result()
            _save_result(conn, q, topics, ...)
```

**Файлы:** `classifier/runner.py` — новый режим `--workers N`
**Ожидание:** 17с → **8-10с/вопрос** (GPU RTX 3050 4GB, Qwen 14B ≈ 9GB quantized — 2 параллельных запроса реалистичны)
**Итого:** ~7.4 дней → **~3.5-4 дня**

### 1.2 Retry-логика при ошибках Ollama

**Проблема:** В `classifier/local_llm.py` при любой ошибке LLM вопрос навсегда помечается как failed (строки 124-126). Ollama иногда падает по таймауту или OOM — это transient-ошибки.

**Решение:** 3 повтора с задержкой (2с, 5с, 15с):
```python
for attempt in range(3):
    result = ollama.chat(...)
    if result:
        return result
    time.sleep([2, 5, 15][attempt])
return None
```

**Файлы:** `classifier/local_llm.py`
**Ожидание:** Снижение failed с ~5-10% до ~1-2%

### 1.3 Оптимизация SQL-запроса при старте

**Проблема:** `database/db.py` строки 164-201 — `get_unclassified_questions()` загружает ВСЕ 37k вопросов в RAM, потом фильтрует в Python.

**Решение:** Один SQL с `NOT IN`:
```sql
SELECT q.id, q.text, q.answer, q.comment
FROM questions q
WHERE q.id NOT IN (
    SELECT DISTINCT question_id FROM question_topics WHERE model_name = ?
)
ORDER BY q.id
```

**Файлы:** `database/db.py`
**Ожидание:** Старт классификации: ~2-3с вместо ~10-15с (при 30k+ уже классифицированных)

## 2. КАЧЕСТВО: повысить надёжность классификации

### 2.1 Проверка few-shot необходимости для Qwen 14B

**Контекст:** Бенчмарк на 50 вопросах показал 88% с few-shot. Но few-shot примеры добавляют ~700 токенов к каждому запросу (замедление ~8-10%).

**Действие:** Запустить бенчмарк Qwen 14B **без** few-shot (`--no-few-shot --twostage --limit 50`) и сравнить accuracy. Если ≥85% — отключить few-shot для ускорения.

**Файлы:** `scripts/benchmark.py` — запустить с `--no-few-shot`
**Потенциал:** Если few-shot можно убрать → ускорение ~10% (17с → 15с)

### 2.2 Валидация на случайной выборке

**Проблема:** Бенчмарк проведён на одном пакете (50 вопросов из одного турнира). Это может не отражать распределение всей базы.

**Действие:** После классификации первых 1000 вопросов:
1. Выбрать 50 случайных из разных пакетов
2. Вручную проверить правильность
3. Посчитать accuracy на реальном распределении
4. Выявить систематические ошибки (bias по категориям)

**Файлы:** Новый скрипт `scripts/validate_sample.py` — случайная выборка + интерфейс ручной проверки

### 2.3 Анализ уверенности (confidence calibration)

**Проблема:** Qwen 14B выдаёт confidence=0.9 почти всегда. Непонятно, стоит ли доверять результатам с conf=0.5 так же, как conf=0.9.

**Действие:** После завершения классификации — построить калибровочную кривую:
- Сгруппировать вопросы по confidence (0.3-0.5, 0.5-0.7, 0.7-0.9, 0.9-1.0)
- Для каждой группы выборочно проверить accuracy
- Если conf=0.5 даёт 80% accuracy, а conf=0.9 даёт 90% — модель плохо откалибрована

**Файлы:** Расширить `scripts/validate_sample.py`

## 3. АНАЛИТИКА: получить actionable инсайты

### 3.1 Расширение аналитических запросов

Текущие запросы в `Analytics/queries.py` и `dashboard/db_queries.py` дают базовую картину. Не хватает:

**Новые запросы:**

| Запрос | Что даёт | SQL |
|--------|----------|-----|
| **Рост/падение категорий** | Какие тематики набирают популярность, какие уходят | Сравнение % за последний год vs предыдущий |
| **Топ-подкатегории по годам** | Динамика внутри категорий | GROUP BY year, subcategory |
| **Авторы × категории** | Какие авторы специализируются на каких темах | JOIN packs.authors с question_topics |
| **Сложность × категория** | Какие темы чаще в сложных пакетах | JOIN packs.difficulty с question_topics |
| **Редкие темы** | Подкатегории с <1% вопросов — потенциальные «сюрпризы» | COUNT < threshold |
| **Парные категории** | Какие тематики чаще встречаются вместе (у вопросов с 2 тегами) | Self-join question_topics |

**Файлы:** `Analytics/queries.py`, `dashboard/db_queries.py`

### 3.2 Новые визуализации в дашборде

**Текущие вкладки:** Обзор, Категории, Сравнение моделей, Уверенность, Поиск

**Добавить:**

1. **Вкладка "Тренды"** — отдельная вкладка (сейчас тренды спрятаны внутри "Категории"):
   - Линейный график: рост/падение каждой категории по годам
   - Heatmap: месяц × категория (интенсивность = количество вопросов)

3. **Вкладка "Авторы"**:
   - Топ-авторы по количеству пакетов
   - Тематический профиль автора (radar-chart)
   - «Автор X предпочитает Историю и Литературу»

**Файлы:** `dashboard/app.py`, `dashboard/components.py`, `dashboard/db_queries.py`
